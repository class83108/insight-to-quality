以下是「訂單退款」功能的 6 類 scenario 覆蓋判斷：

---

## 訂單退款 — Feature Scenario 覆蓋率分析

### 1. Happy path — 適用

- 使用者對已完成訂單申請退款，系統建立退款申請
- 審核人員審核通過，系統發起退款至原付款方式
- 退款成功後，訂單狀態更新為「已退款」

### 2. Error / Failure paths — 適用

- 退款申請送出後，審核人員拒絕退款，訂單狀態維持不變
- 發起退款時，金流回呼失敗（如原付款方式已失效），系統應處理並通知
- 外部金流 API 逾時或錯誤，退款狀態應回滾或標記為「待重試」

### 3. Input boundary — 適用

- 申請退款金額為 0（應拒絕）
- 申請退款金額超過原訂單金額（應拒絕）
- 退款原因欄位為空白或超出字數限制
- 退款金額為原訂單金額的完整退款 vs. 部分退款

### 4. Edge cases — 適用

- 對同一筆訂單重複申請退款（已有進行中的申請時，應拒絕）
- 訂單狀態不是「已完成」時申請退款（如：處理中、已取消）
- 退款申請已通過後，再次嘗試操作（冪等保護）
- 退款金流成功但 webhook 延遲到達

### 5. State mutation — 適用

- 審核通過後，退款紀錄寫入 DB，訂單狀態正確更新
- 退款失敗後，訂單狀態不應被改變
- 重複觸發退款回呼（webhook 重送），系統應保持冪等，不重複退款

### 6. Output contract — 適用

- 退款申請建立的回應包含申請編號、狀態、申請時間
- 審核結果通知包含審核人員、結果、時間戳
- 退款完成後，回應包含退款金額、退款管道、預計到帳時間

---

以上 6 類全部適用。請確認這份分析是否符合你的需求，或有需要調整的地方（例如：是否有部分退款的需求？金流整合的具體行為？），確認後我再開始撰寫 .feature 檔。
