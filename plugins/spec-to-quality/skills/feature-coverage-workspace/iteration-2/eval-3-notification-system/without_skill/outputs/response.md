# Feature Scenario 覆蓋率分析：通知推播系統

以下針對 6 類 scenario 類別逐一判斷，請確認後我再開始撰寫 .feature 檔。

---

## 1. Happy Path — 適用

正常流程端到端，涵蓋：

- 使用者設定 email + push 偏好，高優先事件觸發 → 兩個管道同時推播
- 使用者設定 SMS 偏好，低優先事件觸發 → 僅 push（依優先級降級規則）
- 排程延遲發送：事件發生後 30 分鐘才送出通知
- 批次合併：同一使用者在 5 分鐘內累積 3 則通知 → 合併成一則發送

---

## 2. Error / Failure Paths — 適用

- Email 服務商 API 回傳 5xx → fallback 到 push
- Push 服務失敗且 SMS 為備用管道 → 改走 SMS
- 所有管道都失敗 → 記錄失敗狀態，不靜默丟棄
- 排程任務執行時外部依賴不可用 → 重試策略（次數上限、backoff）
- 批次 job 失敗中途 → 部分已送出的不重複發送（冪等保護）

---

## 3. Input Boundary — 適用

- 使用者未設定任何通知偏好 → 系統行為（靜默 or 預設管道）
- 事件優先級為邊界值（最高/最低/無效值）→ 管道選擇正確
- 延遲時間設為 0 → 立即發送
- 批次合併視窗設為極大值（如 24 小時）→ 行為合理
- 通知訊息為空字串或超長文字 → 截斷或拒絕
- 使用者 ID 不存在 → 明確錯誤而非靜默忽略

---

## 4. Edge Cases — 適用

- 同一事件觸發兩次（重複事件）→ 不重複發送（冪等）
- 使用者在排程期間更改通知偏好 → 以發送時的偏好為準 or 以設定時的偏好為準（需確認語義）
- 同一使用者同時有 1000 則待批次通知 → 合併邏輯不崩潰
- 不同使用者的通知不互相干擾（隔離性）
- 管道優先級相同時的決策行為

---

## 5. State Mutation — 適用

- 通知發送成功後，狀態正確寫入（`sent_at`、`channel` 欄位）
- 排程任務建立後，DB 有正確的 `scheduled_at` 記錄
- 批次合併後，原始通知標記為 `merged`，不再獨立發送
- Re-run 同一批次 job → 冪等，不重複寫入或重複發送
- 偏好更新後，舊偏好正確覆蓋（不殘留）

---

## 6. Output Contract — 適用

- 每則通知的 payload 包含必要欄位（`user_id`、`channel`、`content`、`priority`、`sent_at`）
- Email 通知格式符合 RFC 5322（有 subject、from、to）
- Push 通知 payload 符合 FCM / APNs schema
- 批次合併後的通知，`merged_count` 欄位正確
- API 回傳的通知狀態列表，分頁格式符合 schema

---

## 待確認事項

在你確認前，我需要釐清幾個語義問題：

1. **管道降級規則**：當使用者選的管道失敗時，是 fallback 到其他管道，還是直接失敗？
2. **偏好更新 vs 排程**：排程期間使用者改了偏好，用「設定時」還是「發送時」的偏好？
3. **批次合併視窗**：是固定視窗（e.g., 每 5 分鐘一批）還是滾動視窗？
4. **優先級與管道的對應規則**：高優先 → 強制 SMS？還是使用者偏好優先？

請確認以上分析並回答這幾個問題，我再開始撰寫 .feature 檔。
