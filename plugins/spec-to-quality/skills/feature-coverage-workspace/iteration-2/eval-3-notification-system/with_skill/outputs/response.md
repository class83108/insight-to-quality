## Step 2: 覆蓋率分析表

**功能描述**：通知推播系統，包含：
- 使用者設定通知偏好（email / push / SMS）
- 系統根據事件優先級決定推播管道
- 支援排程延遲發送
- 支援批次合併

| # | 類別 | 適用？ | 具體 Scenario 方向 | 不適用原因 |
|---|------|:------:|-------------------|-----------|
| 1 | **Happy path** | 是 | (1) 使用者設定偏好後，系統依優先級選擇正確管道並成功發送；(2) 排程延遲通知在指定時間後成功送出；(3) 多筆同類通知在批次視窗內合併成一筆發送 | — |
| 2 | **Error / Failure paths** | 是 | (1) Email provider 暫時失敗 → fallback 到 push；(2) 所有管道均失敗 → 記錄失敗狀態並重試；(3) 使用者關閉所有管道偏好 → 通知靜默丟棄或記錄；(4) 排程任務 worker crash → 重啟後不重複發送 | — |
| 3 | **Input boundary** | 是 | (1) 偏好設定為空（未選任何管道）；(2) 事件優先級為邊界值（最低 / 最高等級）；(3) 批次視窗設定為 0 或負數；(4) 延遲時間超大值（例如 365 天後） | — |
| 4 | **Edge cases** | 是 | (1) 同一使用者在批次視窗內收到 1 筆通知（不合併，直接送出）；(2) 使用者在通知排程期間修改偏好 → 以哪個偏好為準？；(3) 同一事件觸發多次（冪等性）；(4) 高優先級事件覆蓋批次合併邏輯（不等待視窗，立即發送） | — |
| 5 | **State mutation** | 是 | (1) 通知狀態正確轉移（pending → sent / failed）；(2) 批次合併後原始通知記錄正確標記；(3) 重複執行排程不產生重複通知（DB-level 冪等）；(4) 使用者偏好更新後舊的排程通知如何處理（是否取消重建） | — |
| 6 | **Output contract** | 是 | (1) 推播 payload 格式符合各管道 schema（email HTML / push JSON / SMS 純文字）；(2) 批次合併後的通知訊息格式正確（例如「您有 3 則新訊息」）；(3) 通知送出事件的 audit log 欄位完整（timestamp, channel, status, retry_count） | — |
