## Feature Scenario 覆蓋率分析：訂單退款

### 1. Happy path — **適用**
- 使用者對已完成訂單申請退款，審核通過，退款成功回到原付款方式
- 可能需要拆成兩個 scenario：「申請退款」與「審核通過後退款執行」

### 2. Error / Failure paths — **適用**
- 申請退款時訂單狀態不符（例如訂單尚未完成、已退款、已取消）
- 審核拒絕 → 退款不執行，通知使用者
- 退款回原付款方式失敗（外部金流 API 異常）→ 應有重試或補償機制
- 審核流程逾時（若有 SLA 設計）

### 3. Input boundary — **適用**
- 退款金額為零或負數
- 退款金額超過原始訂單金額
- 退款原因為空白或超過長度限制
- 重複對同一筆訂單申請退款

### 4. Edge cases — **適用**
- 訂單部分退款（若支援）vs 全額退款
- 同一筆訂單同時有多個退款申請（併發）
- 原付款方式已失效（例如信用卡過期、帳戶被關閉）
- 訂單金額為 0 元的退款申請

### 5. State mutation — **適用**
- 退款申請建立後，訂單狀態應正確轉換（例如 `completed` → `refund_pending`）
- 審核通過後狀態轉換：`refund_pending` → `refunded`
- 退款執行冪等性：重複觸發退款不應重複扣款
- 退款失敗後狀態回滾是否正確

### 6. Output contract — **適用**
- 退款申請的回應包含必要欄位：退款申請 ID、狀態、申請時間
- 審核結果通知格式正確（若有 event/webhook/email）
- 退款完成後的訂單查詢結果包含退款資訊

## 待確認的需求問題

在你確認前，我也有幾個問題影響 scenario 設計：

1. 是否支援**部分退款**（退款金額 < 訂單金額）？
2. 審核流程是**人工審核**還是系統自動審核（或兩者都有）？
3. 退款金流失敗時的處理策略：重試幾次？還是需要人工介入？
4. 申請退款是否有**時間限制**（例如完成後 30 天內才能申請）？
