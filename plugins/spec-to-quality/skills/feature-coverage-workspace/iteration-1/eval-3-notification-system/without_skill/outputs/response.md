# Feature Scenario 覆蓋率分析：通知推播系統

針對你描述的功能（使用者通知偏好設定、事件優先級決定推播管道、排程延遲發送、批次合併），以下是 6 類 scenario 的覆蓋判斷：

---

## 1. Happy Path — 正常流程端到端

**適用**

- 使用者設定偏好 email + push，觸發高優先級事件，兩個管道都收到通知
- 使用者設定偏好 SMS，觸發低優先級事件，系統選擇 SMS 管道發送
- 使用者設定延遲 30 分鐘，事件觸發後於 30 分鐘後發送
- 多個低優先級事件在批次視窗內觸發，合併為一則通知發送

---

## 2. Error / Failure Paths — 外部依賴失敗、例外處理

**適用**

- Email provider 發送失敗，系統 fallback 到 push 或記錄失敗狀態
- SMS gateway 逾時，通知標記為 failed，不重複發送
- 排程任務執行時找不到對應使用者（帳號已刪除），跳過並不報錯
- 推播 token 過期（push notification），發送失敗並清除無效 token

---

## 3. Input Boundary — 空值、超大值、格式錯誤

**適用**

- 使用者未設定任何偏好管道，事件觸發時系統的行為（不發送 / 使用預設管道？）
- 延遲時間設為 0，視同立即發送
- 延遲時間設為極大值（如 30 天），系統是否有上限限制
- 批次合併視窗設為 0 秒，每個事件單獨發送

**不適用**

- email 格式驗證 — 假設偏好設定時已驗證，推播層不重複驗證

---

## 4. Edge Cases — 特殊情境

**適用**

- 使用者在排程延遲期間更改偏好，排程發送時使用新偏好還是舊偏好
- 同一事件重複觸發（冪等性），批次視窗內不應產生重複通知
- 使用者同時啟用所有管道（email + push + SMS），三個管道都收到
- 高優先級事件在批次視窗內觸發，應立即發送而非等待批次

---

## 5. State Mutation — DB / 檔案寫入正確性、冪等性

**適用**

- 通知發送成功後，狀態更新為 `sent`，不會再次發送
- 排程任務被執行兩次（重啟、crash recovery），通知不重複發送
- 批次合併後，原始的多個事件記錄標記為已合併，不會個別發送
- 使用者更新偏好設定，舊的排程通知行為定義明確

---

## 6. Output Contract — 回傳格式符合 schema、欄位完整性

**適用**

- Email 通知包含必要欄位：收件人、主旨、事件摘要
- Push notification payload 符合 FCM/APNs 格式規範
- 批次通知合併後，訊息內容正確列出多個事件摘要
- 排程發送的通知，`sent_at` timestamp 在預期延遲時間之後

---

## 待確認的設計問題（影響 scenario 邊界）

在你確認覆蓋率分析前，有幾個設計決策需要釐清：

1. **Fallback 策略**：某管道發送失敗時，是否 fallback 到其他管道，還是直接標記失敗？
2. **排程時的偏好快照**：延遲發送應使用「事件觸發當下」還是「實際發送當下」的使用者偏好？
3. **批次視窗定義**：批次視窗是固定時間（如每 5 分鐘一批），還是事件觸發後的滑動視窗？
4. **優先級對管道的映射規則**：高優先級 → 強制 push/SMS？低優先級 → 僅 email？規則是否可設定？

請確認這 6 類覆蓋是否符合你的需求，以及上面的設計問題，我再開始撰寫 `.feature` 檔。
